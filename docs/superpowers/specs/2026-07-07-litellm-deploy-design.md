# LiteLLM Gateway — Design

Status: draft · 2026-07-07 · Owner: Artem
Milestone: M5, sub-project 1 of 4 (see [decomposition](../../../../agentops-engine/docs/superpowers/specs/2026-07-07-m5-decomposition.md) in `agentops-engine`)

## Context

`secrets/litellm/` today is an empty `.gitkeep` placeholder; no ArgoCD Application deploys anything for it. This sub-project stands up the real thing: a LiteLLM proxy reachable in-cluster at a stable Service address, with an admin API `agentops-engine`'s new backend (sub-project 2) and a human operator can both use to mint virtual keys with hard spend caps. Nothing calls it yet — that's sub-project 2 — so this doc's job is the deploy + the concrete contract (endpoint, auth, model-alias naming) the next sub-project builds against, same relationship M4 sub-project 1 (observability stack) had to sub-project 2 (OTel instrumentation).

## Goal

`kubectl get applications -n argocd` shows `litellm` `Synced`/`Healthy`; the proxy is reachable at `litellm.platform.svc.cluster.local:4000` with a working OpenAI-compatible `/chat/completions` route to a real z.ai GLM model under the alias `zai-glm-4.6`; the master key can create scoped virtual keys with `max_budget` via `/key/generate`.

## Non-goals

- **Creating the actual per-role/per-product virtual keys.** This sub-project only ensures the master key and admin API work; minting a virtual key for a specific product×role is either a manual operator `curl` against the admin API or later automation (`ConfigSync`-style, M7+ territory) — out of scope here.
- **Public exposure.** Unlike Grafana/Temporal Web, this proxy holds the master key's blast radius (it can mint/revoke every virtual key) behind a purely internal Service + `.lab` Ingress. No public host. Named risk below if that's ever reconsidered.
- **Redis caching.** `cache: true` + Redis is a real LiteLLM feature (response caching, rate-limit coordination across replicas) but there's exactly one replica and no caching need yet — YAGNI until a concrete latency/cost reason shows up.
- **Any provider besides z.ai.** OpenRouter, direct Anthropic/OpenAI — future `model_list` entries once there's a concrete need; the contracts-side design (sub-project 2) treats `backend: 'litellm'` as provider-agnostic specifically so adding these later is a `model_list` entry here, not an engine code change.
- **Subscription rate-window awareness.** Sub-project 4, entirely orthogonal (that's the CLI/subscription lane, this proxy only ever sees API-key traffic).

## Design

Same pattern as every other platform component: one ArgoCD `Application` + Kustomize `helmCharts:` inflator + `values.yaml` under `clusters/ops/platform/litellm/`, registered in `clusters/ops/kustomization.yaml`, landing in the existing shared `platform` namespace (same reasoning as the LGTM stack: a core platform service, no PKI/DNS-style isolation need).

```
clusters/ops/platform/litellm/
  application.yaml
  kustomization.yaml
  values.yaml
  db-bootstrap-job.yaml
  secret-generator.yaml
secrets/litellm/
  litellm-credentials.enc.yaml       # masterkey, LITELLM_SALT_KEY — generated by this sub-project
  litellm-db-credentials.enc.yaml    # username, password — generated by this sub-project
  litellm-provider-keys.enc.yaml     # ZAI_API_KEY — operator-supplied, placeholder here
```

### Chart

`oci://ghcr.io/berriai/litellm-helm`, version `1.91.0` (confirmed via `helm pull`/`kustomize build --enable-helm` against this exact version — the chart is OCI-only, no `https://` repo alternative exists). Bitnami is pulled in only as an *optional* dependency (`postgresql`/`redis` subcharts, both gated behind `condition:` fields) — since both `db.deployStandalone` and `redis.enabled` are `false` here, neither subchart is invoked, so this doesn't reopen the Bitnami-avoidance decision the Postgres/Grafana components already made.

### Database

`db.useExisting: true`, `db.deployStandalone: false`, pointed at the existing shared Postgres (`postgres-postgresql.platform.svc.cluster.local`, per ARCHITECTURE.md §5.2 "shared Postgres instance, separate DB" — same instance Temporal already uses), database `litellm` (the chart's own default value for `db.database`).

**LiteLLM gets its own Postgres role (`litellm`), not the shared `temporal` role Temporal/visibility already use.** This isn't a least-privilege policy this repo has enforced elsewhere — it hasn't; `temporal_visibility` shares the same `temporal` role. It's forced by a narrower constraint: minting a *new* role+password end-to-end (generate a password, `sops --encrypt` it with the recipient key already in `.sops.yaml`, write a bootstrap Job that authenticates as the existing `temporal` role via its existing secret) needs only the **public** age key already committed to this repo. Reusing the `temporal` role would mean copying its real password into a second secret file, which needs *decrypting* `secrets/postgres/postgres-credentials.enc.yaml` — the private age key, which by design (ARCHITECTURE.md §5.1) exists only in the ArgoCD namespace, never available to whoever is authoring this change. A fresh role sidesteps that boundary entirely and is arguably the better call anyway (least-privilege is already the pattern for GitHub tokens and backend auth secrets elsewhere in this system).

The existing `postgres-initdb` ConfigMap approach (`10-temporal-visibility.sql`) can't be reused: those scripts only run against an *empty* data directory on first initialization, and this Postgres instance is already running. Same problem ARCHITECTURE.md's `temporal-namespace-job.yaml` solves for Temporal namespaces — an idempotent Job, run as an ArgoCD `PreSync` hook (`sync-wave: "-1"`, one wave before the chart's own migration Job, which needs the `litellm` role/database to already exist):

```sh
psql -h postgres-postgresql.platform.svc.cluster.local -U temporal -d temporal <<'SQL'
DO $do$
BEGIN
  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'litellm') THEN
    EXECUTE format('CREATE ROLE litellm WITH LOGIN PASSWORD %L', :'pw');
  END IF;
END
$do$;
SELECT 'CREATE DATABASE litellm OWNER litellm'
WHERE NOT EXISTS (SELECT FROM pg_database WHERE datname = 'litellm')\gexec
SQL
```

`PGPASSWORD` (to connect as `temporal`) comes from the existing `postgres-credentials` Secret via `secretKeyRef` at pod runtime; the new role's password comes from the new `litellm-db-credentials` Secret the same way — the Job wires both in as env vars and never needs either value supplied by whoever authors the manifest. The `:'pw'` psql-variable substitution (fed via `-v pw="$LITELLM_DB_PASSWORD"`) plus `format(..., %L, ...)` avoids building the `CREATE ROLE` statement by naive string concatenation.

### Secrets

Three separate files under `secrets/litellm/`, split by who can produce the value and how it's consumed:

- **`litellm-credentials.enc.yaml`** (`masterkey`, `LITELLM_SALT_KEY`) — both freshly generated (`openssl rand`), both values only this sub-project needs to invent, no external dependency. `masterkey` must start with `sk-` per LiteLLM's own convention. `LITELLM_SALT_KEY` is set now, on a fresh deployment with no stored virtual keys yet, specifically because LiteLLM cannot rotate it later without breaking every previously-issued key's stored credentials — better to set it before any real key exists than to retrofit it.
- **`litellm-db-credentials.enc.yaml`** (`username: litellm`, `password: <generated>`) — as above.
- **`litellm-provider-keys.enc.yaml`** (`ZAI_API_KEY: CHANGEME`) — the one value this sub-project genuinely cannot produce; real z.ai API key is an operator step, same shape as every other provider credential in this repo (`claude-credentials`, `pi-credentials`, per-project GitHub tokens).

`masterkey`/`LITELLM_SALT_KEY` reach the container two ways: `masterkeySecretName`/`masterkeySecretKey` (the chart's own single-key `secretKeyRef` field, → `PROXY_MASTER_KEY` env) for the master key, and `environmentSecrets: [litellm-credentials, litellm-provider-keys]` (→ `envFrom: secretRef`, projects every key in both secrets verbatim as env vars) for `LITELLM_SALT_KEY` and `ZAI_API_KEY` — which is why those two secrets' keys are named exactly as the env vars LiteLLM/its config expect, while `litellm-db-credentials`' keys (`username`/`password`) stay lowercase since the chart maps them through `db.secret.usernameKey`/`passwordKey` instead of a blanket `envFrom`.

### Model routing

One `model_list` entry for now:

```yaml
model_list:
  - model_name: zai-glm-4.6
    litellm_params:
      model: zai/glm-4.6
      api_key: os.environ/ZAI_API_KEY
```

`zai/` is LiteLLM's own built-in provider prefix for Z.AI (confirmed against LiteLLM's provider docs) — no custom `api_base`/`custom_llm_provider` plumbing needed. **`model_name` (`zai-glm-4.6`), not the `zai/glm-4.6` provider string, is the contract sub-project 2 calls this proxy with.** This is a deliberate indirection: `agentops-engine`'s `ModelRef.model` field for a `litellm`-routed stage should be the LiteLLM-side alias, never the raw provider-qualified string — that keeps provider-swap (z.ai → OpenRouter → direct Anthropic, per the non-goals above) a `model_list` edit here, with zero `agentops-engine` change, matching the whole point of fronting these calls with LiteLLM instead of calling z.ai directly.

### Networking

`service.type: ClusterIP`, port `4000` (chart default) — reachable in-cluster at `litellm.platform.svc.cluster.local:4000`, the concrete endpoint sub-project 2's backend targets. One internal Ingress at `litellm.lab` (no public host) so an operator can reach the admin UI (`/ui`) from an admin machine through step-ca, same bare `.lab`-only pattern Temporal Web used before its public host was added — deliberately *not* mirroring that public-host precedent here, since this proxy's master key controls every virtual key's budget, a meaningfully larger blast radius than Temporal Web's read-mostly workflow visibility.

## Testing strategy

`kustomize build --enable-helm` against the real pinned chart version, asserted not to error — verified locally this pass (helm/kustomize confirmed installed, chart pulled and rendered successfully with `db.useExisting`/`redis.enabled: false` and a placeholder `model_list`). No cluster access from this sandbox: `Synced`/`Healthy` status, the bootstrap Job actually creating the `litellm` role against the live Postgres, and a real end-to-end call to z.ac's GLM model through the proxy are **not** verified here — operator follow-up, called out explicitly rather than implied by a passing render check, same posture the observability-stack design took for its own six components.

## Named risks

- **Two Postgres roles now exist on one instance with no formal access boundary between them** (`temporal` can still see/alter the `litellm` database and vice versa unless explicitly revoked — Postgres role creation doesn't isolate by default). Not tightened here; flagged as a real gap rather than assumed away, since "least privilege" was the stated motivation for the separate role in the first place.
- **`LITELLM_SALT_KEY` is unrecoverable if lost** (SOPS-encrypted in git, so the usual "lost secret" story is "restore from git + the age key," not a bespoke risk) — but if ever set incorrectly, the fix is destroy-and-recreate every virtual key, not a config edit. Getting it right before any real key exists (this sub-project) is the mitigation, not a runtime safeguard.
- **The bootstrap Job's `PreSync`/`sync-wave: "-1"` ordering relative to the chart's own migration Job (`PreSync`, wave `0`) is inferred from ArgoCD's documented hook-phase + sync-wave semantics, not verified against a live sync** — no cluster access this pass. If ArgoCD doesn't respect sync-wave ordering *within* the same hook phase the way documented, the migration Job could race the role/database's creation; an operator should confirm ordering in the Argo CD UI on first sync.

## Package/file summary

- **New:** `clusters/ops/platform/litellm/{application,kustomization,values,db-bootstrap-job,secret-generator}.yaml`.
- **New:** `secrets/litellm/{litellm-credentials,litellm-db-credentials}.enc.yaml` (real, generated this pass) and `secrets/litellm/litellm-provider-keys.enc.yaml` (placeholder `ZAI_API_KEY`, real value is an operator step).
- **Changed:** `clusters/ops/kustomization.yaml` (one new resource), `docs/DEPLOY.md` (new phase covering the `ZAI_API_KEY` runbook step and the in-cluster endpoint contract for `agentops-engine`).

## Open questions carried forward

- Whether the `litellm`/`temporal` Postgres roles should eventually get explicit `REVOKE`-based isolation — named above, not resolved here.
- Whether a second `model_list` provider (OpenRouter, direct Anthropic) is needed before sub-project 3 demonstrates the gate, or z.ai alone is sufficient — current assumption is z.ai alone is sufficient, since the gate only needs *one* backend distinct from `claude`.
- Real subscription/spend numbers for the first virtual key's `max_budget` (used to deliberately trip the cap in sub-project 3) — deferred to that sub-project's design; this doc only guarantees the admin API can create one.
